The Approach

Free model thinks. Agent builds.

Most developers open their agent immediately when they get a task. That's the expensive mistake. Agents are powerful but priced accordingly. Don't pay for thinking when free models think just as well.

1. Get task
→
2. Craft prompt (free AI)
→
3. Run agent
→
4. Review diff
Open ChatGPT or Claude.ai (free tier). Describe the task. Ask it to help you write a clear, structured prompt for your coding agent. Refine until it's airtight. Then paste it into Claude Code. This single habit cuts agent token usage by 40–60%.

The free model handles ambiguity. The agent handles execution. When the agent gets a precise prompt, it produces precise output — no corrections, no back and forth, no wasted tokens re-reading a growing history.

Review the diff, not the codebase. If the diff matches the prompt and the tests pass, you're done.

Token Cost Formula

The math that changes everything

Total = S × N(N+1) / 2
Messages (N) Total tokens (~500 tok/msg) vs. 5 messages
5 7,500 —
10 27,500 3.7×
20 105,000 14×
30 232,500 31×

Message 30 costs 31× more than message 1 — not because it's longer, but because the model re-reads the full conversation context on every message. N is the variable with the most leverage. Keep sessions short. Start fresh. Batch everything.

10 Token Efficiency Habits

Keep N small. Keep prompts sharp.

  • 01
    Edit your prompt — don't send a follow-up
    Click edit on your last message, fix it, regenerate. Follow-ups stack context. Edits replace it.
  • 02
    Start fresh every 15–20 messages
    Ask Claude to summarize → copy it → new chat → paste as first message. Saves 40–60% on long sessions.
  • 03
    Batch your questions
    Three questions in one message = one context load. Three separate messages = three context loads.
  • 04
    Use Projects for recurring files
    Files added to a Project are cached. The same schema or spec in multiple chats gets tokenized every time without it.
  • 05
    Set Memory and Preferences once
    Your role, style, and coding preferences set at account level. Never spend 3–5 messages on setup again.
  • 06
    Turn off unused features
    Web search and connectors add tokens to every response. Only enable them when you need them.
  • 07
    Use the right model
    Haiku for simple tasks. Sonnet for real dev work. Opus for architecture and complex reasoning only.
  • 08
    Spread work across the day
    Claude uses a rolling 5-hour window. Split heavy sessions into morning / afternoon / evening.
  • 09
    Work off-peak
    Peak hours: 5AM–11AM PT weekdays. Same work runs faster and cheaper at evenings or weekends.
  • 10
    Enable overage as a safety net
    A mid-session cutoff costs more in reconstruction time than the overage. Set a monthly cap and move on.
Developer bonus — Habit #11: Drop claude.md into your project root. Claude reads it automatically every session. Never explain your stack, conventions, or preferences in chat again. That's 200–500 tokens back, every single session, for the life of the project.
Model Guide

Match the model to the task.

Haiku → Formatting, grammar, renaming, commit messages, quick Q&A, seed data 1×
Sonnet → Feature implementation, debugging, tests, code review, refactoring ~5×
Opus → Architecture decisions, complex multi-file refactors, security audit ~15×

Start cheap. Escalate only when the task genuinely requires it — not as a default, not "to be safe." Sonnet handles almost all real development work. Opus is for decisions where being wrong is expensive.

TDD Agent Loop

Tests are the contract. Agents implement against them.

Without tests, agents drift. With tests, you review a diff — not a codebase.

Task arrives ↓ Craft prompt with free model (5–10 min) ↓ Agent writes failing tests first ↓ Agent implements to pass the tests ↓ Run tests ↓ Pass → Review diff → Done Fail → Agent self-corrects (max 2 retries) ↓ Still failing? → You step in, adjust the prompt, retry
Token cost comparison
Path ~Tokens
Write 10 tests + implement + review ~7,000
No tests + 1 correction round ~14,000
No tests + 2 correction rounds ~34,000
Quick Prompts

Copy, fill in the blanks, send.

Write failing tests first
Before writing any implementation, write failing tests for {{feature}}. Cover: happy path, empty input, invalid input, edge cases. Use {{Jest / Vitest}}. Tests must FAIL before any implementation exists. Do not write implementation code yet.
Implement to pass the tests
Write the implementation for {{feature}} that passes all the tests you just wrote. Only create or modify files needed. Do not change the tests. Show me the diff only — not the full files.
Summarize this session
Summarize this session before we continue: 1. What we built or changed (file names) 2. What's in progress and what step we're on 3. Decisions made and why 4. What's next Under 150 words. I'll start a new chat with this as context.
